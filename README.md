# Антифрод-профилирование клиентов

Проект исследования создан в рамках дипломной работы по теме "Проектирование системы антифрод-анализа банковских операций на основе агрегированных данных". 

Проект строит клиентскую витрину по транзакциям IEEE-CIS Fraud Detection и проверяет несколько ML-подходов для ранжирования клиентов по риску мошенничества. Основной сценарий - формирование приоритетной очереди для ручной проверки.
Для воспроизведения процесса нужны исходные CSV, ноутбуки и папка `ML` с экспериментами.

## Что делает проект

1. Проводит первичный EDA по транзакциям.
2. Собирает клиентский профиль из транзакций и идентификационных данных.
3. Обучает базовые модели на профильной витрине.
4. Последовательно проверяет CatBoost, LightGBM, XGBoost, абляционный анализ признаков и устойчивость лучшей модели.
5. Формирует выгрузки клиентов с высоким риском и пакет для экспертной проверки.

## Данные

Для полного воспроизведения нужны файлы из соревнования IEEE-CIS Fraud Detection:

- `train_transaction.csv`
- `train_identity.csv`

В текущем корне уже используется:

- `train_transaction.csv` - исходные транзакции;
- `client_profile_v1_0_shuffled.csv` - готовая профильная витрина, от которой стартуют ML-эксперименты.

Если воспроизводить процесс с нуля, положите `train_transaction.csv` и `train_identity.csv` в рабочую папку проекта или в Google Drive по пути:


## Воспроизведение процесса

### 1. Первичный EDA (опционально)

Откройте и выполните:

```text
first_EDA.ipynb
```

Этот ноутбук нужен для разведочного анализа исходных транзакций: размерности, пропуски, распределение класса мошенничества, связь транзакций с таблицей `train_identity`.

### 2. Сборка клиентской витрины

Откройте и выполните:

```text
transaction_to_profile_converter.ipynb
```

На входе ожидаются `train_transaction.csv` и `train_identity.csv`. На выходе формируются клиентские агрегаты, агрегаты по идентификационным данным и финальная профильная таблица. Основной итоговый файл для ML-части:

```text
client_profile_v1_0_shuffled.csv
```

### 3. Базовые модели: ML v1.0

Запустите:

```text
ML/1.0/client_profile_v1_0_baseline.ipynb
```

Проверяется модель логистической регрессии и модель случайного леса. Цель шага - доказать, что профильная витрина содержит ML-сигнал. Лучшая базовая модель на тестовой выборке: случайный лес с `PR-AUC ≈ 0.4210` и `ROC-AUC ≈ 0.8810`.

Описание результатов:

```text
ML/1.0/V1_0_RESULTS.md
```

### 4. CatBoost: ML v1.1

Запустите:

```text
ML/1.1/client_profile_v1_1_catboost.ipynb
ML/1.1/client_profile_v1_1_catboost_longrun.ipynb
```

Здесь проверяются CatBoost, ExtraTrees и более длительный запуск CatBoost. Лучший длительный запуск CatBoost получил на тестовой выборке `PR-AUC ≈ 0.4708`.

Ключевые артефакты:

```text
ML/1.1/results_default/
ML/1.1/results_longrung/
ML/1.1/V1_1_RESULTS.md
```

### 5. Абляционный анализ признаков: ML v1.2

Запустите:

```text
ML/1.2/client_profile_v1_2_ablation.ipynb
```

Этот этап проверяет вклад групп признаков: идентификационные признаки, почтовые домены, суммы операций, активность во времени, прокси-признаки рискованных потоков и разреженные идентификационные поля. Основной вывод: набор `full_base` остается лучшим, блок идентификационных признаков критически важен, а ультра-разреженные идентификационные поля не дают стабильного прироста.

Ключевые артефакты:

```text
ML/1.2/results_ablation/
ML/1.2/V1_2_RESULTS.md
```

### 6. Сравнение бустингов: ML v1.3

Запустите:

```text
ML/1.3/client_profile_v1_3_boosting_benchmark.ipynb
```

Сравниваются CatBoost, LightGBM и XGBoost на том же наборе признаков. В этом эксперименте `lightgbm_default` стал новым сильным кандидатом: на тестовой выборке `PR-AUC ≈ 0.4779`.

Ключевые артефакты:

```text
ML/1.3/results_differet_boostings/
ML/1.3/V1_3_RESULTS.md
```

### 7. Финальный LightGBM: ML v1.4

Основной ноутбук:

```text
ML/1.4/client_profile_v1_4_lightgbm_confirm.ipynb
```

Для Kaggle есть отдельная версия:

```text
ML/1.4/client_profile_v1_4_lightgbm_confirm_kaggle.ipynb
```

Этот этап подтверждает LightGBM и считает рабочие точки ROC/FPR, пороговые таблицы, качество в верхней части ранжирования и Top-5000 клиентов. Основной кандидат после этого шага:

```text
lgbm_auc_unweighted_seed2026
```

Итоговое качество на тестовой выборке:

```text
PR-AUC ≈ 0.4967
ROC-AUC ≈ 0.8987
```

Ключевые артефакты:

```text
ML/1.4/results_lightgbm_confirm/
ML/1.4/V1_4_RESULTS.md
```

Дополнительное сравнение CatBoost и LightGBM по Top-5000:

```text
ML/1.4/client_profile_v1_4_catboost_lightgbm_top5000_compare.ipynb
ML/1.4/results_score_compare/
ML/1.4/V1_4_COMPARE_RESULTS.md
```

### 8. Проверка устойчивости: ML v1.5

Запустите:

```text
ML/1.5/client_profile_v1_5_verification.ipynb
```

Это не поиск новой модели, а проверка выбранного LightGBM из `v1.4`:

- устойчивость к случайным инициализациям;
- доверительные интервалы по бутстрэпу;
- перемешивание целевой переменной;
- приближенное временное разбиение.

Главный вывод: модель `v1.4` устойчива на текущей витрине, но на приближенном временном разбиении качество ниже, поэтому временную устойчивость нужно описывать осторожно.

Ключевые артефакты:

```text
ML/1.5/results_verification/
ML/1.5/V1_5_RESULTS.md
```

### 9. Экспертная проверка: ML v1.6

После получения Top-5000 и сохраненной LightGBM-модели можно сформировать пакет для экспертной оценки:

```bash
python ML/1.6/generate_human_review_pack.py
```

По умолчанию скрипт использует:

```text
client_profile_v1_0_shuffled.csv
ML/1.4/results_lightgbm_confirm/ml_v1_4_top5000_test_clients_by_roc_auc_model.csv
ML/1.4/results_lightgbm_confirm/ml_v1_4_best_by_roc_auc.joblib
```

Результат сохраняется в:

```text
ML/1.6/results_human_review/
```

Основные выходные файлы:

```text
human_review_cases_blinded.csv
human_review_answers.csv
human_review_master_key.csv
human_review_questionnaire.md
human_review_questionnaire.html
human_review_generation_summary.json
```

`human_review_master_key.csv` содержит служебную разметку и должен использоваться только после завершения экспертной оценки.

Итоги экспертной проверки описаны в:

```text
ML/1.6/V1_6_HUMAN_REVIEW_RESULTS.md
```

## Ожидаемая последовательность артефактов

```text
train_transaction.csv + train_identity.csv
    -> client_profile_v1_0_shuffled.csv
    -> ML/1.0 метрики базовых моделей
    -> ML/1.1 результаты CatBoost
    -> ML/1.2 результаты абляционного анализа
    -> ML/1.3 сравнение бустингов
    -> ML/1.4 финальный LightGBM, пакет joblib, выгрузки Top-5000
    -> ML/1.5 отчеты проверки устойчивости

```

## Важные замечания

- `PR-AUC` используется как основная метрика, потому что класс мошенничества редкий.
- `ROC-AUC` и `Recall@FPR` нужны для банковской интерпретации с контролем ложных срабатываний.
- Порог `0.5` не является универсальным рабочим порогом. Порог нужно выбирать под сценарий: верхние k клиентов, ограничение FPR или требуемый баланс точности и полноты.
- Top-5000 по обучающей, валидационной и тестовой частям не является честной тестовой метрикой качества, потому что включает обучающие данные. Для оценки качества используйте тестовую часть и таблицы метрик.
- Экспертная проверка в `v1.6` показывает, что экспертные решения нестабильны; модель корректнее трактовать как инструмент приоритизации и предварительного отбора, а не как замену эксперту.
